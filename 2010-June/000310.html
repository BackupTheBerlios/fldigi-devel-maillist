> Basically, the high level view is:
> 
> 1. Set up the application for localization. This will include:
> integrating gettext (*) in the build system, initialize gettext in the
> code, mark the strings for translation.
> 
> 2. Extract the translatable strings from the code to make them available
> to translators. Generally running a make rule or a script that
> ultimately produces a POT template file (e.g. application.pot).
> Translators work from this file and create their own translations as PO
> files.
> 
> 3. Integrate translations in the code. This will generally mean
> committing the PO files containing the translations (e.g. ca.po, es.po,
> de.po, it.po, etc) and building them, that is, converting them to the
> binary format (MO files) that gettext understands and can load at
> runtime.
> 
> Once 1 and 2 are set up, it is fairly trivial to expose those
> translations in Launchpad (enable translations for the project, choose a
> translation policy -recommended: Restricted-, chose a translation group
> -recommended: Launchpad Translators-, enable automatic template imports
> and automatic translation exports).
> 
> I can provide more details on each step, but I didn't want to give too
> much info in an introductory e-mail to translations :)
> 
> >I contacted the package maintainer of one of the most popular packages,
> >fldigi and asked about his plans - he has people already working on
> >translations, but not using any framework that we might have. So I'm
> >also curious to know how Ubuntu translations might be of help to him.
> >
> 
> I've had a quick look at the fldigi tarball. It seems that the package
> is already set up for translations, although I haven't tested if they
> work.
> 
> What _we_ can do to help them is let them know about the Launchpad
> Translations infrastructure and tell them how they can benefit from it.
> 
> What _they_ can do is to expose their translation templates in Launchpad
> to give them more visibility, use the existing pool of Launchpad
> translators and the nice online translation functionality. Here's an
> example of how an upstream such as Getting Things GNOME uses it:
> 
> https://translations.launchpad.net/gtg/trunk/+pots/gtg
> 
> The important thing to notice here is that they can use Launchpad for
> translations without having to migrate their infrastructure to
> Launchpad. It would be nice and they'd benefit from bzr integration
> features if they did, but it's not a requisite.
> 
> >Here are some lists of the packages that the amateur radio team works
> >with: https://wiki.ubuntu.com/UbuntuHamsPackages
> >
> >Can you point me to some starting information?
>
