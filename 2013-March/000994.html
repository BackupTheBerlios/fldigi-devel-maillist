Excellent question :):)

Why I do not follow the wiki ? I tried, but decided to try
cross-compilation for several reasons (Which have nothing to do with
Mingw):

(1) There is only one PC running Windows at home.
(2) It belongs to my wife, who sometimes uses it! Or the kids watch a
film or play a game !!
(3) It's a slow machine, running an extremely slow antivirus software
which takes hours scanning every bit of code I try to install.
(4) It 's in front of a very uncomfortable chair, in another room than mine.

On the other hand, if I can cross-compile from my usual (and exclusive...) box:
(1) No need to copy the files from one place to the other.
(2) Can cross compile to different targets at the same time.
(3) One machine, saves room, energy (And only one confortable chair is
needed :):) )

>>>
>>> My quick reaction without thinking too much is that it is better to have
>>> a dependency on an actively-maintained external library than it is to
>>> copy library sources into fldigi (or any other program).  When there are
>>> security fixes (or other fixes) in the library, it's then necessary to
>>> import the new version and have an fldigi release (including stable
>>> branches).
>>>
>>> The notion that it's easier to port if internal likely strongly applies
>>> only to windows.  It might make sense to have a windows binary installer
>>> with a bundle, but then the regeneration process would just be a new
>>> bundle, not an fldigi release.  In the unix world, it seems preferred to
>>> just expect the package for the other library to be installed.
>>>
>>> Questions in selection are of course code quality, but also availability
>>> and being mainstream.  In pkgsrc, xmlrpc-c is packaged (at version
>>> 1.16.44), but I don't see xmlrpcpp.  That's not definitive, but it's a
>>> clue that xmlrpc-c is in widespread use and xmlrpcpp is not.
